Clean Code

2018-07-15

The only valid mesurement of code quality : WTFs/Minute.

引言

作为一只经验尚浅的程序猿，我经常会思考如何提升自己的编程水平，写出好的代码。

在日常开发中，我们通常会接触到别人写的代码，如果这部分代码是很复杂，难以理解的，那么维护它的人无疑会带着沉重和悲怆的心情在心里面默默的诅咒。但是，如果这部分代码是清晰简单的，那么维护它的人的心情也会变得顺畅，可见好的代码不仅能够使程序提高效率更能使人舒服。

那么，什么样的代码才是好的代码呢？对于这一点，我的认识是，好的代码必定是反复思考的结果，它不一定很复杂，但应该是逻辑清晰，结构简单的，不仅容易复用而且对未来的扩展有一定的前瞻性。

也因此，我认为一个软件的构建过程应该是这样，在良好的架构下，由众多简单逻辑的小组件构成，这些小组件可以相互组合成更大的组件，同样也是逻辑简单的。不过，事实上，我们经常看到的代码要么是过度设计或者根本没有设计的，它们不仅一开始就复杂，不简洁，当与其他类似模块协同工作后其复杂度更是加倍，难以理解。

那么如何才能写出好的代码呢？
是做出很多酷炫的功能吗？还是写个几万几十万行的代码，等待灵感的出现？肯定这些都不是。

我认为最有效的方式应该是，多参考优秀的代码设计，并在此基础上，设计出满足当前需求的代码。

有时候，在我们思考过后，脑袋里有了思绪，但是无法在代码中实现。就好比中学时期的语文作文，对于同样的题目我绞尽脑汁也想不出满分作文那样优美的金句。出现这个问题时，语文老师经常会给我们推荐优秀的作文或者作文片段，让我们记住并在自己的作文中加以利用。同样对于编程，我们也可以通过学习和记忆优秀的代码片段或者代码结构，提高自己的编程水平。

在这篇文章中，我会不断更新自己见到的优秀的代码设计，多多总结，提高自己的代码质量。
