Good Code

2018-07-15

The only valid mesurement of code quality : WTFs/Minute.

引言

作为一只经验尚浅的程序猿，我经常会思考如何提升自己的编程水平，写出好的代码。

在日常开发中，我们通常会接触到别人写的代码，如果这部分代码是很复杂，难以理解的，那么维护它的人无疑会带着沉重和悲怆的心情在心里面默默的诅咒着。但是，如果这部分代码是清晰简单的，那么维护它的人的心情也会变得顺畅，可见好的代码不仅能够使程序提高效率更能使人舒服。

那么，什么样的代码才是好的代码呢？对于这一点，我的认识是，好的代码必定是反复思考的结果，它不一定很复杂，但应该是逻辑清晰，结构简单的，不仅容易复用而且对未来的扩展有一定的前瞻性。

也因此，我认为一个软件的构建过程应该是这样，在良好的架构下，由众多简单逻辑的小组件构成，这些小组件可以相互组合成更大的组件，同样也是逻辑简单的。不过，事实上，我们经常看到的代码要么是过度设计或者根本没有设计的，它们不仅一开始就复杂，不简洁，当与其他类似模块协同工作后其复杂度更是加倍，难以理解。

如何才能写出好的代码呢？

有时候，在我们思考过后，脑袋里有了思绪，但是无法在代码中实现。就好比中学时期的语文作文，对于同样的题目我绞尽脑汁也想不出满分作文那样优美的金句。出现这个问题时，语文老师经常会给我们推荐优秀的作文或者作文片段，让我们记住并在自己的作文中加以利用。同样对于编程，我们也可以通过学习和记忆优秀的代码设计和代码结构，提高自己的编程水平。

在这篇文章中，我会不断更新自己见到的好的代码，多多总结，提高自己的代码质量。

设计原则(Design Principle)

首先总结下面向对象的设计原则，这些原则在编程实践中起着相当重要的作用。

1. 单一职责原则(Single Responsibility Principle)

一个类应该只有一个发生变化的原因。

如果一个类承担的职责过多，就等于把这些职责耦合在一起了。一个职责的变化可能会削弱或者抑制这个类完成其他职责的能力。这种耦合会导致脆弱的设计，当发生变化时，设计会遭受到意想不到的破坏。而如果想要避免这种现象的发生，就要尽可能的遵守单一职责原则。此原则的核心就是解耦和增强内聚性。

2. 里氏替换原则(Liskov Substitution Principle)

所有引用基类的地方必须能透明地使用其子类的对象。

里氏替换原则包含以下4层含义：

  1. 子类可以实现父类的抽象方法，但是不能覆盖父类的非抽象方法。
  2. 子类中可以增加自己特有的方法。
  3. 当子类覆盖或实现父类的方法时，方法的前置条件（即方法的形参）要比父类方法的输入参数更宽松。
  4. 当子类的方法实现父类的抽象方法时，方法的后置条件（即方法的返回值）要比父类更严格。

3. 依赖倒置原则(Dependence Inversion Principle)

高层次的模块不应该依赖于低层次的模块，他们都应该依赖于抽象。抽象不应该依赖于具体实现，具体实现应该依赖于抽象。

4. 接口隔离原则(Interface Segregation Principle )

客户端不应该依赖它不需要的接口；一个类对另一个类的依赖应该建立在最小的接口上。接口设计应该尽可能精简。

5. 迪米特法则(Law of Demeter)

一个对象应该对其他对象保持最少的了解。每个类尽量减少对其他类的依赖。

6. 开闭原则(Open Closed Principle )

一个软件实体如类、模块和函数应该对扩展开放，对修改关闭。
